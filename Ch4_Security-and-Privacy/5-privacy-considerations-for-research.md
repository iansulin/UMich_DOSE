# 5. Privacy Considerations for Research

Firebase security rules and device identification (Sections 3 and 4) are technical controls. This section addresses **research privacy** — how your Firebase configuration fits into ethical study conduct, participant trust, and institutional requirements.

---

## 5.1 Types of sensitive data collected

The DOSE Watch App collects and transmits several categories of data to Firebase. Understanding sensitivity helps you communicate accurately in consent forms and security plans.

### Data categories

| Category | Examples from DOSE | Typical sensitivity |
|----------|-------------------|---------------------|
| **Health and activity data** | Step counts, heart rate samples | High — may reveal sleep patterns, illness, location inference via activity |
| **Ecological momentary assessment (EMA)** | Survey responses to prompts | High — mood, substance use, symptoms (depends on your survey modules) |
| **Behavioral metadata** | Notification delivery and open timestamps | Medium — reveals daily routines, engagement patterns |
| **Study identifiers** | User ID entered on watch | Medium — linkable to enrollment records if stored separately |

### Combined sensitivity

Individual fields may seem low-risk in isolation. Combined, they can be highly identifying:

**Example — inference from metadata alone:**

| Field | Value | Possible inference |
|-------|-------|-------------------|
| `healthkitdata` timestamps | Activity peaks at 2–3 AM daily | Irregular sleep schedule |
| `notifications.opened` | Consistently 45+ minutes after delivery | Participant may be at work during windows |
| `survey.response` | Low values on weekday mornings | Context-dependent (design, mental health, fatigue) |

A numeric User ID in Firebase may still be linkable to an individual if your enrollment records map codes to names (e.g., `10042 → participant name`). Unless confirmed otherwise by a qualified privacy reviewer, treat Firebase data as identifiable research data.

HealthKit data remains on the device until the app uploads it. Consent materials should describe both on-device storage and subsequent transfer to Firebase.

## 5.2 Participant identifiers and the device ID (User ID)

### How User IDs work in DOSE

- Participants enter a **User ID** once at study start (`ParticipantView`).
- The ID is stored in the device **Keychain** and used as the Firebase path key (`deviceID` in code).
- The app displays a warning: *"the User ID cannot be changed later."*
- The app rejects empty IDs and discourages values like `test` or `demo`.
- The app does **not** auto-generate a UUID for Firebase — see Section 4.2.

### Identifier design recommendations

| Practice | Rationale | Example |
|----------|-----------|---------|
| **Use study-specific codes, not names** | Prevents direct identification in Firebase | `84729103` not `jsmith` |
| **Avoid sequential IDs** | Reduces guessability of Firebase paths | Random 8-digit codes, not `0001`, `0002` |
| **Maintain a separate enrollment key** | Firebase holds codes; identity key held separately with stronger access controls | Firebase: `84729103` / Secure enrollment DB: maps to participant record |
| **Document ID assignment procedure** | Reproducibility and audit | "IDs assigned from pre-generated list at in-person visit" |
| **Plan for ID compromise** | Lost watch may expose ID on screen | Assign a new User ID and re-enroll if compromised |

### What the User ID is NOT

| User ID is… | User ID is not… |
|-------------|-----------------|
| A study participant code | A Firebase login credential |
| Stored on-device in Keychain | Auto-generated by the app |
| Stable for the study duration (by design) | Changeable by the participant after setup |

The User ID appears as plain text in database paths (`/devices/84729103/...`). Anyone with database read access — for example, through misconfigured rules — can view these codes.

### Simulator and test identifiers

In Xcode simulator, IDs are prefixed with `TEST-`:

| Environment | User ID entered | Firebase path key |
|-------------|-----------------|-------------------|
| Physical watch | `84729103` | `84729103` |
| Simulator | `84729103` | `TEST-84729103` |

**Example mistake:** Using real participant IDs in the simulator against a production Firebase project creates `/devices/TEST-84729103/` paths that may confuse analysis pipelines.

**Best practice:** Never enter real participant IDs in simulator testing. Use dedicated test codes (`TEST-99999`) in a development Firebase project.

## 5.3 Data minimization and retention

### Data minimization

Collect and retain only what your study protocol requires.

| Question | Action |
|----------|--------|
| Do you need minute-level heart rate, or are hourly aggregates sufficient? | Adjust HealthKit query windows in app configuration |
| Do you need indefinite storage in Firebase, or only during active collection? | Plan export + deletion |
| Do all team members need Firebase Console access? | Limit to those who require it (Section 5.6) |

**Example retention policy statement (for your data management plan):**

> Active study phase: data retained in Firebase Realtime Database for real-time monitoring and weekly export. Within 30 days of study completion, all participant data exported to institutional secure storage and deleted from Firebase. Backups retained per university policy for 7 years, access restricted to PI and data manager.

### Export before deletion

Before deleting Firebase data at study end:

1. Export all `/devices/` data using Firebase Admin SDK or Console export tools
2. Verify export completeness (record counts, date ranges)
3. Store export in institutional approved storage (not personal Google Drive)
4. Delete Firebase data per protocol
5. Document deletion date and responsible person

Deleting a Firebase project is irreversible. Export all required data first and confirm IRB-approved retention periods before deletion.

### Example export command (technical collaborator — Admin SDK)

```javascript
// Illustrative — run server-side only
const snapshot = await admin.database().ref('devices').once('value');
const allParticipantData = snapshot.val();
// Save to secure institutional storage
```

Studies with frequent HealthKit uploads can accumulate substantial data volume. Monitor storage in the Firebase Console **Usage** tab and plan export intervals accordingly.

## 5.4 Encryption in transit and regulatory context

The DOSE Watch App transmits all Firebase uploads over **HTTPS**, which uses **TLS (Transport Layer Security)** to encrypt data between the Apple Watch and Google's servers. Understanding what this protects—and what it does not—is essential for ethics applications, institutional review, and compliance with health and privacy regulations.

### How data moves from the watch to Firebase

```
Apple Watch  →  HTTPS (TLS-encrypted)  →  Google Firebase Realtime Database
```

Every upload in `FirebaseManager.swift` uses standard `URLSession` HTTP `POST` requests to your Realtime Database URL (`https://YOUR-PROJECT-ID-default-rtdb.firebaseio.com/...`). The `https://` scheme ensures the request body (survey responses, HealthKit records, notification timestamps) is encrypted in transit.

| Stage | Location | Protection |
|-------|----------|------------|
| On device (before upload) | Apple Watch storage / Keychain | Apple platform security; User ID in Keychain |
| In transit | Network path between watch and Google | **TLS encryption (HTTPS)** |
| At rest in Firebase | Google Cloud data centres | Google default encryption at rest; access controlled by security rules (Section 3) |

### What encryption in transit provides

TLS protects against **interception on the network path** between the participant's watch and Firebase. An attacker monitoring an unsecured Wi‑Fi network, for example, should not be able to read upload contents in plain text because the connection is encrypted.

Google's Firebase infrastructure uses industry-standard TLS configurations for Realtime Database REST endpoints. Researchers do not need to configure TLS separately for the DOSE app's default upload mechanism.

### What encryption in transit does not provide

| TLS does not… | What addresses it instead |
|---------------|---------------------------|
| Encrypt data stored in Firebase after arrival | Google encrypts data at rest by default; security rules control who can read it (Section 3) |
| Prevent unauthorized access if security rules are open | Publish restrictive rules before enrollment (Sections 3 and 6) |
| Anonymize or de-identify data | Study-specific User IDs and separate enrollment records (Section 5.2) |
| Satisfy all regulatory requirements on its own | Institutional policies, vendor agreements, and ethics review (below) |

Researchers should describe **both** encryption in transit and access controls in ethics and data-management documentation—not TLS alone.

### Encryption at rest (brief)

Firebase Realtime Database data is stored on Google Cloud infrastructure, which applies **encryption at rest** by default. This is separate from the TLS protection during upload. Access to stored data is governed by Firebase security rules and Firebase Console permissions (Sections 3 and 5.6), not by the participant's watch.

### Regulatory and institutional framing

Health and behavioural data collected by DOSE may be subject to privacy and health-information laws depending on your jurisdiction, institution, funding source, and whether data can be linked to identifiable individuals. The table below is a starting point for discussion with your IRB, privacy office, and legal counsel—it is not legal advice.

| Framework | Typical scope | Relevance to DOSE |
|-----------|---------------|-------------------|
| **HIPAA** (United States) | Protected health information (PHI) held by covered entities and their business associates | Step counts, heart rate, and survey responses may constitute PHI when linked to an individual. Cloud storage of PHI generally requires a **Business Associate Agreement (BAA)** with the vendor and use of HIPAA-eligible Google Cloud / Firebase configurations. |
| **GDPR** (European Union / EEA) | Personal data of individuals in the EU/EEA | Health and behavioural data are often **special category data** (Article 9). Requires a lawful basis, data minimization, and attention to **international transfers** if Firebase region is outside the EU. Google provides a Data Processing Agreement as part of [Google Cloud terms](https://cloud.google.com/terms/data-processing-terms). |
| **UK GDPR / Data Protection Act 2018** | Personal data of UK residents | Similar to EU GDPR; confirm Firebase region and institutional DPA requirements. |
| **PIPEDA** (Canada) | Personal information in commercial and research contexts | Consent, safeguards, and cross-border transfer notification may apply. |
| **Australian Privacy Act / APPs** | Personal information of Australian individuals | Health information may have additional sensitivity; institutional ethics and privacy review required. |
| **Institutional policies** | University hospitals, NHS trusts, provincial health networks, etc. | Many institutions require **vendor security review**, **data processing agreements**, and approved cloud service lists before using Firebase for identifiable or health-related research data. |

### HIPAA considerations (United States)

HIPAA applies when your study team or institution is a **covered entity** or **business associate** handling **PHI**, and the data you store in Firebase meets the definition of PHI (individually identifiable health information).

| Topic | Guidance for research teams |
|-------|----------------------------|
| **Is DOSE data PHI?** | Often **yes** when User IDs can be linked to participants and health metrics (steps, heart rate) are collected. Survey content may also constitute PHI depending on questions asked. |
| **Firebase and HIPAA** | Google Cloud can support HIPAA workloads when configured appropriately, but this requires institutional setup—not default consumer Firebase projects. Consult [Google Cloud HIPAA compliance documentation](https://cloud.google.com/security/compliance/hipaa). |
| **Business Associate Agreement** | If PHI is stored in Firebase, your institution typically needs a **BAA** with Google. This is arranged through institutional IT or legal counsel, not individual researchers. |
| **Minimum necessary** | Align collection with protocol scope (Section 5.3). Document why each data type is needed. |
| **Breach notification** | Unauthorized access to Firebase data (e.g., misconfigured rules) may trigger breach reporting obligations. Maintain a response plan (Section 5.5). |

If your study is **not** subject to HIPAA (e.g., de-identified data under an IRB-approved exemption, or research outside US covered-entity context), document that determination with your privacy office.

### GDPR and international transfer considerations (EU / UK / EEA)

| Topic | Guidance for research teams |
|-------|----------------------------|
| **Lawful basis** | Research may rely on consent, public interest, or other bases depending on national law. Your IRB/ethics approval does not automatically establish GDPR lawful basis—confirm with your DPO or legal team. |
| **Special category data** | Health data and data concerning health often require an Article 9 condition (frequently **explicit consent** for research). |
| **Data location** | Select Firebase Realtime Database **region** at project creation (e.g., `europe-west1` for EU). Document the region in your ethics application. |
| **Processor agreement** | Google acts as a **data processor** when you control study data. Institutional DPAs with Google Cloud may already exist—verify with your institution. |
| **Participant rights** | Depending on jurisdiction, participants may have rights of access, rectification, erasure, and restriction. Plan how requests are handled using your enrollment log and export procedures. |

### What to document for ethics and compliance review

| Document | Include |
|----------|---------|
| **IRB / ethics application** | Data types, cloud provider, Firebase region, encryption in transit (HTTPS/TLS), access controls (Sections 3–4), retention and deletion |
| **Data management plan** | User ID linkage to identity (stored separately), export schedule, who can access Firebase Console |
| **Consent form** | Cloud storage, encryption in transit, study code vs. name, retention period (see example in Section 5.5) |
| **Institutional vendor review** | Firebase / Google Cloud approval status, DPA or BAA if required |
| **Protocol amendment** | Required if region, provider, or data types change |

### Example language for ethics applications (encryption and regulations)

> *Data are transmitted from the Apple Watch to Google Firebase Realtime Database over encrypted connections (HTTPS/TLS). Data are stored in [region] with access restricted by Firebase security rules to authorized research personnel. [Institution name] has [completed / is completing] institutional review of Google Cloud as a data processor. This study [is / is not] subject to HIPAA; [BAA status if applicable].*

Adapt all language with your IRB, privacy office, or Data Protection Officer before submission.

## 5.5 Aligning Firebase setup with IRB/ethics requirements

Institutional Review Boards (IRBs) and ethics committees typically require you to describe how participant data is protected. Use the checklist below when preparing or updating your ethics application.

### Information to disclose to ethics reviewers

| Topic | What to include |
|-------|-----------------|
| **Cloud provider** | Google Firebase Realtime Database |
| **Data types** | HealthKit metrics, survey responses, notification timestamps |
| **Data location** | Firebase region (e.g., United States, European Union) |
| **Identifiers** | Study-specific numeric codes; link to identity stored separately |
| **Access controls** | Security rules summary and User ID assignment approach (Section 4) |
| **Who can access data** | Named roles (PI, coordinator, IT) — not "the research team" vaguely |
| **Retention and deletion** | Timelines and responsible party |
| **Transmission security** | HTTPS/TLS for all uploads from Apple Watch to Firebase (Section 5.4) |
| **Storage security** | Firebase security rules; Google encryption at rest; access limited to named roles |
| **Regulatory context** | Applicable frameworks (e.g., HIPAA, GDPR, institutional policy); BAA or DPA status if required |
| **Risks** | Residual cloud storage risks despite encryption in transit, encryption at rest, and access controls |
| **Participant disclosure** | Consent form language about cloud storage (see example below) |

### Example consent language (adapt with your IRB's guidance)

> *Study data collected by the DOSE Watch App (including activity and heart rate information, survey answers, and app usage timestamps) will be transmitted over encrypted connections (HTTPS/TLS) and stored in a secure cloud database (Google Firebase) in [region]. Access to this database is restricted to authorized research personnel. Your name is not stored in the cloud database; instead, a study code is used. Data will be retained for [X years] and then deleted, as described in the study privacy plan.*

IRB requirements for consent language vary by institution. Submit draft text to your ethics board before enrollment rather than adopting the example above without review.

### Mapping technical controls to ethics documentation

| Ethics document section | Technical control reference |
|-------------------------|----------------------------|
| Data security plan | Sections 3–5.4 of this guide; rule version in use; regulatory determinations |
| Data flow diagram | Watch → HTTPS/TLS → Firebase `/devices/{User ID}/` |
| Breach response plan | Firebase Console audit logs; contact Google / institutional security |
| Protocol amendments | Required if you change cloud provider, region, or data types |

### When to submit a protocol amendment

Submit an amendment if you:

- Move from a development Firebase project to production
- Change Firebase geographic region
- Add new data types to uploads
- Change security rules or User ID format
- Share Firebase access with new external collaborators

## 5.6 Access control: who should have Firebase Console access

Firebase Console access is separate from security rules. Console users with sufficient Google Cloud IAM roles can **read all data regardless of rules** and **change rules themselves**.

### Firebase / Google Cloud roles (simplified)

| Role | Can view data | Can edit rules | Can delete project |
|------|---------------|----------------|--------------------|
| **Viewer** | Yes (via Console) | No | No |
| **Editor** | Yes | Yes | No |
| **Owner** | Yes | Yes | Yes |

### Recommended access model for research studies

| Person | Recommended access | Why |
|--------|-------------------|-----|
| Principal Investigator | Owner (or institutional admin) | Accountability |
| Study coordinator | Viewer OR no Console access | May not need raw data access if exports are provided |
| Data analyst | No Console access; receives exported datasets | Minimizes exposure |
| IT / developer (setup phase) | Editor during setup; downgrade to Viewer or remove after | Reduces ongoing risk |
| Student research assistant | Generally no Console access | High turnover; training burden |

### Access management practices

- [ ] Use an **institutional Google account** or Google Workspace group (e.g., `dose-study-admins@university.edu`) as project Owner — not a personal Gmail that may be deactivated when a student graduates
- [ ] Review access list **quarterly** and remove departed team members promptly
- [ ] Enable **2-factor authentication (2FA)** on all accounts with Firebase access
- [ ] Log access reviews in study documentation (date, reviewer, changes made)

**Example access review log:**

| Date | Reviewer | Action |
|------|----------|--------|
| 2025-06-01 | Dr. Smith | Removed Editor access for graduated RA Jane Doe |
| 2025-09-01 | Dr. Smith | Added Viewer access for data manager Bob Lee |

Do not share a single Google account among team members. Each person with Firebase access should use an individual account to support auditability.

For institutions with advanced compliance requirements, [Cloud Audit Logs](https://cloud.google.com/logging/docs/audit) record Console access and configuration changes. Your IT team can assist with enabling this for your Firebase project.

## Privacy checklist before enrollment

- [ ] Consent form describes cloud storage, data types, and encryption in transit (HTTPS/TLS)
- [ ] Firebase region documented and aligned with ethics application
- [ ] Applicable regulatory requirements identified (e.g., HIPAA, GDPR, institutional policy)
- [ ] BAA, DPA, or institutional vendor approval obtained if required (Section 5.4)
- [ ] User IDs are non-identifying and non-sequential
- [ ] Enrollment key (code → identity) stored separately with access controls
- [ ] Firebase region matches ethics application / participant location requirements
- [ ] Retention and deletion schedule documented and approved
- [ ] Firebase Console access limited to named individuals
- [ ] Test data isolated from production data
- [ ] Security rules and User ID practices documented in data management plan
